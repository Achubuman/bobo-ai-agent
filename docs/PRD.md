## 1. Problem Statement

辦公室團購的發起流程由 15–20 分鐘的純手工勞動組成：手動查店、截圖菜單、建 Google 表單、逐一催單、最後手算誰欠誰多少。這個痛點每週在每個辦公室群組重複 2–3 次，造成「願意揪的人越來越少、最後大家都外送獨食」的惡性循環。

**誰受影響**：需要發起揪團的辦公室員工（團主），以及被動等開團卻填單體驗差的同事（跟單者）。
**不解決的代價**：團主意願下降 → 辦公室凝聚力流失；個人外送費用增加（無法湊免外送門檻）；行政效率無端耗損。

## 2. Goals

| # | 目標 | 衡量方式 | 目標值（v1 上線後 4 週） | 現況 |
|---|---|---|---|---|
| G1 | 縮短開團完整流程時間 | 從「說出想揪」到「表單貼出群組」耗時 | < 3 分鐘（baseline：15 分鐘） | 🟡 Demo 佐證流程已縮短，絕對耗時需上線量測 |
| G2 | 減少對帳糾紛 | 每場團購後「金額不對」的投訴或追問 | 目標 0 件（完全由確定性程式計算） | ✅ Demo 可佐證：Balance Check = 0 |
| G3 | 提高開團頻率 | 同一群組每週開團次數 | 提升 30% | ⬜ 需上線量測 |
| G4 | 使用者滿意度 | 填單體驗 CSAT（5 分制） | ≥ 4.2 分 | ⬜ 需上線量測 |

> **說明**：填單率（多少人選擇點餐）反映成員意願，不是產品品質指標，故不列入 Goals。波波的目標是降低「想揪但嫌麻煩」的門檻，而非創造點餐需求。

## 3. Non-Goals（v1 明確不做）

| # | 不做的項目 | 原因 |
|---|---|---|

## 4. User Stories

### 使用者角色定義

| 角色 | 說明 |
|---|---|
| **團主** | 在 LINE 群組發起開團的人，需要承擔全部流程負擔 |
| **跟單者** | 群組成員，只需填單，希望流程快且不用記密碼 |

### 4.1 F1 吃啥喝啥（探店推薦）

- 作為**團主**，我想對波波說「今天想揪飲料」，讓波波推薦 2–3 間附近店家（含均價、門檻、招牌），這樣我不用自己 Google，可以直接選。
- 作為**跟單者**，我想在群組看到店家推薦，可以直接按喜歡的選項投票，這樣大家共同決定要去哪。

### 4.2 F2 一鍵開團（菜單解析）

- 作為**團主**，我想把菜單圖片傳給波波，讓波波自動解析品項、規格選項，並在貼出前讓我確認，這樣我不用手動建表單。
- 作為**團主**，我想在 OCR 確認步驟能編輯錯誤的品項或價格，這樣不會讓跟單者填到錯的選項。
- 作為**團主**，我希望波波支援飲料、便當、下午茶等不同類別的菜單，這樣我不需要依食物類別換工具。
- 作為**跟單者**，我想在表單裡看到糖度、冰量等對應規格的下拉選單，這樣填單快速不會搞錯。
- 作為**團主**，我想在確認步驟把不想開放的品項或選項「隱藏／刪除」（例如太細的加料、店家今天沒有的品項），讓表單乾淨、跟單者不會選到不該選的（OCR 給草稿、團主做減法）。
- 作為**團主**，遇到「時價／未標價」品項時，我希望被要求先做決定（我來填價／標現場結算不分帳／隱藏），這樣不會有沒有價格的品項混進自動分帳、把帳算錯。
- 作為**團主**，對於「套餐／加購」這種需搭配主餐的項目，我希望表單清楚標示為「加購」並以加價（＋$）呈現、依附在主餐底下，這樣跟單者不會誤點成單點。
- 作為**團主**，對於 AI「推理生成」的選項（甜度／加料／套餐內容，非菜單明寫），我希望它們預設可編輯、不被當成定案，這樣即使 AI 在高信心下填錯，我也能在發單前修掉。

### 4.3 F3 快樂夥伴（開團文案）

- 作為**團主**，我想讓波波自動生成一則菜鳥開團風格的開團文案，包含截單時間，這樣我不用自己想怎麼說話。
- 作為**團主**，我想按「換一個」讓波波換一則不同梗路線的文案，這樣可以挑自己喜歡的語氣貼出去。

### 4.4 F4 叮咚提醒（截單與取餐提醒）

- 作為**團主**，我想在截單前 30 分鐘讓波波自動 tag 未填者，這樣我不需要一個一個點名催促。
- 作為**團主**，我想在截單時收到訂單彙整清單，這樣我可以立刻整理後傳給店家。
- 作為**跟單者**（請假者），我不希望收到波波的提醒訊息，這樣不會被不相關的通知騷擾。
- 作為**跟單者**，我想在餐到時看到取餐清單（姓名＋品項＋規格），這樣我可以快速找到自己的餐點。

### 4.5 F5 算錢天才（計費對帳）

- 作為**團主**，我想選擇四種計費模式（請客／自費／半請補助型／半請一人一份型），讓波波自動計算每人應付金額，這樣我不用自己開 Excel。
- 作為**團主**，我想看到「誰欠我多少、誰多付了多少」的對帳表，這樣我可以直接截圖傳給大家。
- 作為**跟單者**（多付者），我希望多付的錢自動儲值到下一團，這樣不會每次都要找零。

## 6. Success Metrics

> **佐證欄**：區分「Demo 現在就能佐證」與「需上線才能量測」，避免把目標值誤讀為已達成數字。

### 6.1 Leading Indicators（上線後 1–2 週可觀察）

| 指標 | 定義 | 目標值 | 量測方式 | 佐證 |
|---|---|---|---|---|
| **開團完成率** | 從波波收到「想揪」到表單貼出群組的 session 完成率 | ≥ 80% | sessions 表：status = completed ÷ 全部 initiated | 需上線量測 |
| **OCR 確認通過率** | 團主第一次確認（未編輯直接送出）的比例 | ≥ 85% | OCR confirm events：直接確認 ÷ 全部確認 | 需上線量測（品質基礎見 `OCR_testing_and_findings.md`）|
| **填單完成時間** | 跟單者從開啟表單到送出的平均時間 | < 90 秒 | LIFF 表單 open → submit 時間差 | 需上線量測 |
| **文案換款率** | 團主按「換一個」的比例（太高代表初始文案不夠好） | < 30% | 換款 postback 事件 ÷ 文案生成次數 | 需上線量測 |

### 6.2 Lagging Indicators（上線後 4 週可觀察）

| 指標 | 定義 | 目標值 | 佐證 |
|---|---|---|---|
| **截單前填單率** | 截單時已填單人數 ÷ 群組活躍成員 | ≥ 75% | 需上線量測 |
| **開團頻率成長** | 相同群組每週開團次數（vs. 使用前 baseline） | +30% | 需上線量測 |
| **對帳糾紛率** | 每場團購後追問金額的訊息數量 | 0 件 | ✅ Demo 可佐證：確定性計算 Balance Check = 0 |
| **跟單者 CSAT** | 填單後送出時的 1–5 分滿意度評分 | ≥ 4.2 | 需上線量測 |
| **波波留存率** | 使用 3 次以上的群組比例（at week 4） | ≥ 60% | 需上線量測 |

> **不追蹤填單率**：填單率是成員意願指標，非產品品質指標。若某次開團只有 3 人點餐，代表只有 3 人想點，不代表波波有問題。

## 7. Non-Functional Requirements

| 類別 | 需求 | 標準 |
|---|---|---|
| **回應時間** | LINE 訊息回覆 | < 3 秒（P99） |
| **OCR 串流起始** | 圖片收到 → 畫面開始有反應（首個 token）| < 3 秒（Claude 實測 1.5–2.6s）|
| **OCR 完整解析** | 圖片收到 → 完整確認卡片 | 依菜單大小，中位數約 30s；大菜單（55–65 品項）約 **35–70s+**（實測 Gemini 2.5 @16384：金茶伍 33s、雀飲 41s、彤露 72s）。**不設 <8s 硬指標**——N 個品項逐一生成有物理下限。首個品項可見：Claude 從 ~2s 起逐段串流；Gemini 2.5 先 thinking ~25–60s 才吐字（早期回饋差）。量測方法與逐張數據見 `OCR_testing_and_findings.md` §7 |
| **可用性** | 服務 uptime | ≥ 99% |
| **OCR 信心治理** | 不以單一信心分數自動跳過人工確認；分軸治理：價格類嚴管（null 或低信心一律強制確認）、推理選項可編輯不阻擋；needs_review 保留 1-tap 快速核對 | 強制 |
| **OCR 量化評測** | 以離線腳本比對辨識輸出與 ground truth，量測召回率／價格正確率／幻覺率（詳見 `OCR_testing_and_findings.md`）| 召回率 ≥95%、價格正確率 ≥95%、幻覺率 0% |
| **大菜單可 scale** | 輸出長度不隨品項數線性爆量撞 token 上限 | 以 store_defaults 繼承＋分類分組壓縮輸出（80 項菜單約 10× 壓縮）；截斷時 repairArray 還原至最後完整品項 |
| **安全性** | LINE userId 驗證，防止冒用他人身分填單 | 透過 LIFF.getProfile() 驗證 |
| **資料保存** | 訂單與對帳資料 | 保存 90 天 |
