# 菜單 OCR 測試與發現

> 對象：波波 F2 菜單解析的核心提示詞與辨識流程（`../menu_ocr_prompt.md`）。
> 撰寫角度：AI PM —— 不只驗證「能不能跑」，而是驗證「能不能放心交給團主用」。
> 這份是 `PRD.md` 中 OCR 相關決策（分軸治理、needs_review 門檻、store_defaults 壓縮）的**證據來源**。
> 測試素材：10 張真實菜單（`../test_menus/`），涵蓋飲料／正餐／套餐／手寫改價／模糊小圖／非菜單負向案例。

---

## 1. 為什麼要測這個（問題定義）

波波的開團流程裡，OCR 是**信任的起點**：團主拍一張菜單，系統要把它變成可以點餐的結構化表單。這一步只要錯一個價格或漏一個品項，後面整團的金額就會錯，團主要自己善後。所以辨識品質直接決定產品「敢不敢自動發表單」，也決定了 `needs_review`（人工確認）這道防線該設多嚴。

測試要回答三個產品問題：

1. **準不準**：價格、品項、分類抓得對嗎？（決定能不能信）
2. **聰不聰明**：會不會依菜單類型自動生出合理的客製欄位（甜度／飯量／辣度／套餐）？（這版的差異化價值）
3. **誠不誠實**：辨識沒把握時，會不會老實標成低信心、把球丟回給團主？（決定 `needs_review` 機制是否可靠）

## 2. 測試範圍

| 測什麼 | 在範圍內 | 不在範圍內 |
|---|---|---|
| 對單張菜單的辨識正確率 | ✅ | 端到端開團流程 |

| 輸出 JSON 格式合法性 | ✅ | 多語言菜單（首版只測 zh-TW） |

## 3. 測試案例設計（每案：輸入 → 預期 → 通過標準）

### TC-01　純飲料店（標準清晰）
- **輸入**：手搖飲菜單，含中／大杯不同價、珍珠椰果加料。
- **預期**：每品項依尺寸拆成獨立物件；自動生成「甜度」「冰量」；有珍珠則生成「加料」。
- **通過**：① 品項與價格正確 ② 尺寸拆開無合併 ③ 甜度／冰量存在 ④ `overall_confidence ≥ 0.95`。

### TC-02　便當／正餐店
- **輸入**：便當店菜單（雞腿飯、排骨飯…）。
- **預期**：自動生成「飯量」；若菜單有辣度則生成「辣度」。
- **通過**：① 主餐品項與價格正確 ② 有「飯量」欄位 ③ 未捏造菜單沒有的配菜。

### TC-03　下午茶／點心（炸物、雞蛋糕、紅豆餅）
- **輸入**：鹹酥雞或點心類菜單。
- **預期**：自動生成「口味／調味」（胡椒／梅粉／加辣等）。
- **通過**：① 調味欄位來自菜單實際文字（菜單沒寫才套預設）② 價格正確。

### TC-04　套餐／組合餐
- **輸入**：含「A套餐／超值組／定食」的菜單。
- **預期**：`custom_options` 生成「主餐／附餐／飲料」；菜單沒寫細節時填「請洽店家確認」。
- **通過**：① 套餐被識別 ② 不亂編套餐內容 ③ 缺資訊時誠實標註而非幻覺。

### TC-05　混搭店（飲料＋餐點同張）
- **輸入**：一張菜單同時有飲料與餐點。
- **預期**：依各品項類型分別套用對應客製欄位（飲料給甜度冰量、餐點給飯量）。
- **通過**：分類正確，欄位不互相污染。

### TC-06　邊界：模糊／反光／陰影
- **輸入**：刻意拍糊或反光的菜單。
- **預期**：可疑價格／品項進入 `low_confidence_items`，`overall_confidence` 下降，備註用 ⚠️ 提醒團主。
- **通過**：**寧可標低信心、不要硬猜**。模糊處被誠實標註＝通過；硬編一個錯價＝失敗。

### TC-07　邊界：手寫／貼紙改價
- **輸入**：手寫菜單或貼紙蓋住舊價的菜單。
- **預期**：手寫品項信心降低並標註；新舊價並存時以可見新價為準或標為低信心。
- **通過**：不把被蓋住的舊價當成有效價格。

### TC-08　格式約束（每案都驗）
- **預期**：① 純 JSON 陣列、無 Markdown 圍欄與前言 ② 價格為純整數（無「$」「元」）③ `scan_summary` 是陣列第一個元素 ④ 全程 zh-TW。
- **通過**：可直接被 JSON 解析；schema 驗證通過。

### TC-09　負向／抗幻覺
- **輸入**：把一張**非菜單**圖片（如風景照）丟進去。
- **預期**：不應捏造品項；應回低信心或明確說明無法辨識。
- **通過**：沒有幻覺出不存在的餐點。

## 4. 評分標準與 KPI

針對每張測試菜單，人工建立「標準答案（ground truth）」後比對：

| 指標 | 定義 | 目標 |
|---|---|---|
| 品項召回率 Recall | 正確抓到的品項數 ÷ 菜單實際品項數 | ≥ 95% |
| 價格正確率 | 價格完全正確的品項數 ÷ 總品項數 | ≥ 95% |
| 尺寸拆分正確率 | 多尺寸品項正確拆成獨立物件的比例 | 100% |
| 客製欄位合理率 | 自動生成欄位符合菜單類型且無幻覺的比例 | ≥ 90% |
| 信心校準 | 實際辨識錯誤的品項，有被標進低信心的比例 | ≥ 80% |
| 格式合法率 | 輸出可被 JSON 解析且通過 schema 驗證 | 100% |
| 幻覺率 | 捏造菜單上不存在品項／選項的比例 | 0% |

> **PM 觀點**：**信心校準** 與 **幻覺率** 比單純準確率更關鍵。95% 準確率但會默默編價，比 90% 準確率但會誠實標記不確定，對「敢自動發單」這個產品決策更危險。

## 6. 逐張驗證重點（節錄）

**雀飲（純飲料，清晰）── TC-01**
- 尺寸拆分正確：M／L 雙價品項全部拆成獨立物件，無一合併。
- 用菜單實際選項而非預設模板：甜度／冰量抓菜單原文，沒套通用預設。
- 加料來源正確：「免費加蘆薈／含蘆薈／含梅子」正確帶入而非亂加珍珠。
- 誠實標記：季節限定未標價 → `price: null` 並標低信心。

**小木屋鬆餅（套餐＋鬆餅＋茶＋炸物）── TC-04 / TC-05**
- 套餐規則正確：各套餐生成「附餐＋飲料」欄位，並把「加價升級、每份限一套」寫進備註。
- 混搭店分類乾淨：飲料才有甜度／冰量，鬆餅與炸物不會被誤加飲料欄位。

**蔥油餅（手寫改價）── TC-07（最能展示價值）**
- 正確處理紅筆改價：印刷價 30 被改為 35 → 採可見新價 35，**同時標進低信心並備註「原價 30 改為 35」**。
- 把店家備註轉成可點選項：「不要胡椒鹽請提早告知」→ 生成「原味(含胡椒鹽)／不要胡椒鹽」。

**模糊小圖 ×2 ── TC-06（抗幻覺，最重要的安全測試）**
- 寧可認輸、不硬猜：兩張都是低解析度小圖，整體信心壓到 0.42 / 0.38，建議團主重拍。
- 只保留可確認資訊：逐項價格一律 `null`，不亂填數字，幻覺率＝0。

> **PM 重點**：模糊圖的「失敗」其實是產品的成功。一個會在模糊圖上硬編價格的 OCR，比一個老實說「我看不清、請重拍」的 OCR 危險得多。

## 8. 三個 Finding（AI PM 對 LLM 不確定性的防護網）

真實菜單踩出三個盲點。三者的共同主線：**模型遇到沒被明確指示的視覺訊號，預設行為都是「照抄／硬填」。AI PM 的工作，就是把這些盲點一個個變成明確規則，並用 needs_review 兜住還沒被規則覆蓋的長尾。**

### Finding 1：高信心仍可能幻覺 → 分軸治理

**案例**：佳福川味牛肉麵（**整體信心 0.94、圖片清晰**）。模型在湯餃的**加購**欄位，輸出了菜單上**不存在**的「整份加／加半份」，正確應為「加牛肉（+50）／加牛肚（+40）」。**這不是看不清造成的錯，是清晰菜單上的幻覺。**

**根因**：OCR 輸出有兩種，信心分數只管得到其中一種。

| 輸出類型 | 例子 | 信心分數管得到嗎？ |
|---|---|---|
| **照抄型**（transcribed） | 品項名、印刷清楚的價格 | ✅ 與「辨識清晰度」相關，高信心＝可信 |
| **推理生成型**（inferred） | `custom_options`（甜度／加料／套餐內容） | ❌ 模型依菜單類型「推論」出來的，圖再清楚也可能編錯 |

本案錯在「加購」這個**推理生成欄位**，所以 0.94 高信心保護不到它。**關鍵洞察：信心分數衡量「看不看得清」，但幻覺發生在「推理生成」欄位 —— 兩者是不同軸，不能用同一個門檻管。**

**對策（分層，已落地於提示詞與辨識流程）**
1. **欄位分級信任（最關鍵）**：每個選項標來源 `source: "menu"`（菜單明寫）或 `"inferred"`（推論）。前端把 inferred 欄位標成「建議選項、可編輯」，**不論信心多高都不自動視為定案**。
2. **以「錯誤成本」決定門檻，而非單一信心數字**：價格錯＝動到錢 → 任何 `price=null` 或價格被標低信心，**一律強制團主確認**；選項標籤錯＝可自行改、不涉金額 → **不阻擋發單，但永遠可編輯**。
3. **needs_review 不被任何分數「完全跳過」**：至少保留 **1-tap 團主快速核對**（5 秒掃一眼就能抓到「加牛肚」這種錯）。
4. **先做信心校準**：信心是模型自報，需用標註好答案的測試集量測「自報 0.95 是否真的 ≈ 95% 正確」。目前 10 張不足以統計，需擴充標註集。

> **回答「幾分才不用團主確認？」**：以目前資料，不建議用任何單一分數自動跳過確認。這次的錯不是「看不清」（0.94 其實很清楚），把門檻提到 0.97 也擋不掉。正確作法是**分軸治理**：價格走嚴格門檻＋校準、推理選項一律可編輯不阻擋、並保留 1-tap 快速確認。

### Finding 2：手寫「刪除」未被處理 → 編輯語意要有加有減

**案例**：蔥油餅手寫單（TC-07）。菜單上「捲餅 一張」「豬肉餡餅 半張」的尺寸被藍筆**劃掉（刪除線）**，但 OCR 仍輸出「捲餅(一張)」「豬肉餡餅(半張)」，**把已刪除的尺寸照抄進品名**。

**根因**：手寫編輯有「加」也有「減」，原規則只做了「加」。

| 編輯方向 | 例子 | 原狀態 |
|---|---|---|
| **加**（改價、加註） | 紅筆把 30 改 35 | ✅ 已處理（採新價＋提醒） |
| **減**（劃掉、刪除） | 藍筆把尺寸／品項槓掉 | ❌ 無規則 → 模型把劃掉的字當有效文字照抄 |

**對策（已落地）**：新增「刪除」雙向規則 —— 整個品項被劃掉→**不輸出**；只有尺寸／某選項被劃掉→品名／選項**移除被劃部分**（「捲餅 一張」劃掉→輸出「捲餅」）；被劃掉處標進 `low_confidence_items` 提醒團主確認。

### Finding 3：大菜單輸出超過 token 上限被截斷 → store_defaults 壓縮

**案例**：金茶伍（手搖飲，**80＋ 品項**）。即時解析時輸出 JSON 在中途**突然斷掉、沒有收尾**，前端報「無法解析」。**這不是辨識錯，是輸出長度超過 `max_tokens` 被截斷。**

**根因**：schema 太囉嗦，無法 scale 到大菜單。每一杯飲料都**完整重複**甜度（7 項）＋冰量（4 項）＋加料（6 項）三個陣列 → 80 杯 × 重複陣列 → 輸出 token 暴增撞上限。**輸出長度沒有隨菜單變大而被設計過。**

**對策（分層，已落地）**
1. **截斷救援**：`repairArray()` 在輸出被切斷時，還原到最後一個完整品項並補收尾，顯示「已還原 N 項＋提示」，而不是整批失敗。
2. **壓縮 schema（治本）**：
   - **store_defaults 繼承**：全店一致的甜度／冰量／加料只在 `scan_summary` 放一份，品項只列差異、其餘繼承。
   - **分類分組＋省略空欄位**：以分類為單位輸出，無差異的品項只剩 `name`+`price`。
   - 合計效果：80 項大菜單輸出 token 從 ~1.6 萬降到 ~1.2–1.5 千（約 **10×**），同時砍成本、截斷風險、時間三者。
3. **串流漸進顯示**：邊收邊用 `repairArray` 渲染已完成品項。總時間不變，但**第一批品項數秒內就出現、不會「卡著像當掉」**，即使未跑完也看得到部分結果。
4. **誠實的下限**：N 個品項就是要逐一生成，輸出有物理下限。最終是「壓 token（治本）＋串流（體感）＋快取（避免重複）」三者並用，而非單靠一招。

> **PM 收斂**：能用真實大菜單踩到「輸出長度有天花板、schema 要考慮會不會 scale」這個雷，正是堅持用真實菜單、而非乾淨樣本測試的價值。

## 9. 依測試反饋的迭代紀錄

第一輪測試後，依實際使用情境回饋修正辨識規則，這段本身就是 AI PM 的「測試 → 定義問題 → 改規格 → 驗證」閉環。

| # | 原問題 | 修正後規則 | 影響 |
|---|---|---|---|
| 1 | 「免費加蘆薈／含蘆薈」被當唯一加料，鎖死其他加料 | 區分『隨附/優惠』與『可加料清單』，隨附料標明後仍附上其他可加料 | 點餐自由度正確 |
| 2 | 單一價格品項保守標低信心 | 價格對齊 L 欄即標 (L)，不扣信心 | 雀飲信心 0.93→0.97 |
| 3 | 套餐「2選1」明列兩款仍標待確認 | 「N選1 視為完整集合」規則 | 小木屋信心 0.96→0.98 |
| 4 | 手寫改價大幅拉低整體信心 | 拆開「辨識清晰度」與「需人工確認」：字跡清楚就維持高信心，只列清單提醒 | 蔥油餅信心 0.78→0.92 |

> **設計原則**：**信心分數只反映「看不看得清」，不反映「需不需要確認」。**「需確認」用低信心『清單』表達，「看不清」才降『整體分數』。兩者分離後，門檻才好訂。

## 10. 測完得到的產品結論

1. **`null` 價格是合法狀態**：缺價（季節限定）與不可辨識（模糊圖）都需要 `price: null`；資料模型與分帳都要能承載，否則會在自動分帳時出錯。
2. **needs_review 門檻有依據了**：建議 `overall_confidence < 0.90` 一律強制人工確認、禁止自動發單；`< 0.50`（模糊圖）直接回「請重拍菜單」而不進開單流程。
3. **手寫改價／刪除是真實高頻情境**：把「新舊價並存 → 採新價＋強制 review」「劃掉 → 移除」寫成正式規格。
4. **輸出要能 scale**：大菜單以 store_defaults 壓縮，並保留截斷救援與串流，作為對 LLM 輸出不確定性的常備防護。

這些結論已回饋進 `PRD.md` 的 F2 需求與 §7 非功能需求（分軸治理、時價三選一、推理欄位可編輯、大菜單可 scale）。
